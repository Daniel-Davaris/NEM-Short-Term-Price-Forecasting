## Core System Architecture

This is a **5-stage, spike-aware ensemble** electricity price forecaster:

### Stage Pipeline
1. **Data Ingestion** (1_Dataset) → Raw AEMO + weather → Processed parquets
2. **Feature Engineering** (2_Features_build) → 200–400+ features across 8 groups
3. **Feature Selection** (3_Features_select) → MI ranking, CV best-K, deduplication
4. **Model Training** (4_Model) → 6 components × 96 horizons = 672 models
5. **Evaluation** (Later/) → Backtesting, metrics, visualization

### Key Constraints
- **Resolution**: 30-min intervals (aggregated from 5-min NEM data)
- **Horizon**: 96 steps = 48 hours ahead
- **States**: NSW, QLD, VIC, SA (typically focus on NSW)
- **Training Window**: 2019–2026 (~50k time steps)
- **Assumptions**: X_train and y_train are pre-aligned by design (no index intersection needed)

---

## Critical Feature: Predispatch RRP ⭐

**NEVER allow predispatch forecasts (h1_forecast, h24_forecast, h96_forecast) to be dropped from feature selection.**

- **Why**: Predispatch RRP is AEMO's published forecast of prices for next 48h; it embeds systemic supply/demand info
- **Risk**: MI-based ranking ignores conditional importance; predispatch has low marginal MI but high conditional importance
- **Fix**: Force-keep via `MUST_KEEP_FEATURES` env variable in 3_Features_select/4_feature_selection.ipynb
- **Impact**: Dropping predispatch causes ~10%+ accuracy loss

**Related Issue**: P1 (High Priority) in REPOSITORY_CONTEXT.md

## Model Architecture: 6-Component Ensemble

For each horizon h ∈ {1..96}:

| Component | Objective | Target | Use Case | Exists If |
|-----------|-----------|--------|----------|-----------|
| **Base L1** | L1 (MAE) | asinh(min(y_h, p97)) | Median prediction, clipped to remove spikes | Always |
| **Base L2** | L2 (MSE) | asinh(y_h) | Mean prediction, uncapped | Always |
| **Spike Clf** | Binary classification | (y_h > $150) | Detect high-price events | ≥20 spikes in train |
| **Spike Reg** | Regression | asinh(y_h), full range | Specialized predictions for extreme prices | ≥20 spikes in train |
| **Spike Q** | Quantile regression | asinh(y_h), full range | Confidence intervals (q95) for spikes | ≥20 spikes in train |
| **Dip Clf** | Binary classification | (y_h < $0) | Detect negative prices | ≥20 dips in train |
| **Dip Reg** | Regression | asinh(y_h), full range | Specialized for dips | ≥20 dips in train |

**Sample Weights**: 
- Base models: `recency_weight × np.where(y_h > $150, 3.0, 1.0)` (spike upweight)
- Spike models: `recency_weight × 10.0` (high priority for extreme events)

**Prediction Policy** (Not yet implemented; see Issue P5):
- Hard gate: Use spike classifier to choose base (normal) vs. spike reg (extreme)
- Soft blend: Weighted average of components

## Data Validation Rules

When working with X_train, y_train, X_validate, y_validate, X_test, y_test:

✅ **ALWAYS check**:
```python
assert X_train.shape[0] == y_train.shape[0], "Shape mismatch X vs y"
assert (X_train.index == y_train.index).all(), "Index mismatch"
assert X_train.isnull().sum().sum() == 0, "NaNs in features"
assert y_train.isnull().sum().sum() == 0, "NaNs in target"
```

✅ **Understand alignment assumption**:
- X_train and y_train are pre-aligned in 2_Features_build and 3_Features_select stages
- By the time they reach 4_Model, they already share identical indices
- No index intersection needed (unlike defensive code you may see)

✅ **Data shapes** (typical):
- X_train: (50k, 180)  # 50k samples, 180 features
- y_train: (50k, 96)   # 50k samples, 96 forecast horizons
- Features: float32, values ~N(0,1) after scaling
- Target: float (RRP in $/MWh, raw, not transformed)

## Feature Engineering Hierarchy

When discussing or adding features, reference these groups in order:

1. **Time Features** (Always keep): hour_sin, hour_cos, dow, is_holiday, is_peak
2. **Predispatch** (Critical): h1_forecast, ..., h96_forecast ← Force-keep
3. **Price Lags** (Important): rrp_lag_1, rrp_lag_48, rrp_mean_daily
4. **System Stress** (Important): reserve_margin, demand_surprise, thermal_utilization
5. **Weather** (Supporting): temp, cdd, hdd, interactions
6. **Supply Curve** (Domain-specific): bid_curve_slope, equilibrium_quantity
7. **Interactions** (Advanced): cross-products for conditional effects

## When Discussing Code Changes

1. **Clarify the impact**: How many models affected? (Answer: typically 672 if it's in the training loop)
2. **Reference the stage**: Which of the 5 stages? (e.g., "This is 3_Features_select, stage 3")
3. **Check for spike implications**: Does it affect spike detection or weighting?
4. **Document the WHY**: Not just "change X to Y", but "because..."
5. **Link to known issues**: Does it address P1–P5? Reference REPOSITORY_CONTEXT.md

### Example
**Good**: "In 3_Features_select/4_feature_selection.ipynb, force-keep predispatch RRP (addresses P1) by adding MUST_KEEP_FEATURES = ['h1_forecast', ..., 'h96_forecast']. This ensures the critical exogenous signal isn't dropped by MI ranking, preventing ~10% accuracy loss."

**Avoid**: "Change the feature selection code to keep more features."

## Common Prompts & Expected Responses

### Prompt 1: "Why do we need separate spike models?"
**Expected Response**: Reference the bimodal price distribution (normal prices $20–$100 vs. spikes >$150). A single model can't optimize for both (high-variance spikes + low-bias median). Separate models via hard gating achieve both.

### Prompt 2: "Should I remove feature X because it has low MI?"
**Expected Response**: Check if X is in MUST_KEEP_FEATURES (calendar, predispatch). Low marginal MI ≠ low conditional importance. Cross-reference with spike-aware MI ranking (P2 fix). If X isn't essential, it's safe to remove.

### Prompt 3: "Can I change the feature selection CV window?"
**Expected Response**: Current window is 2019–2023 (pre-crisis), which misses post-2022 NEM regime. This is Issue P4 (Medium Priority). Changing to 2026 or rolling split is recommended; impacts all 672 models. Coordinate with REPOSITORY_CONTEXT.md update.

### Prompt 4: "How do I validate the training output?"
**Expected Response**: Load models from joblib. For each horizon h, check:
- 7 model objects returned (base_l1, base_l2, spike_clf, spike_reg, spike_q, dip_clf, dip_reg)
- Non-None objects indicate sufficient spike/dip samples in training
- Run predictions on validation set; compare to baseline (MAE, RMSE, spike detection AUC)
